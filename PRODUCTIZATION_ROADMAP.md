# 얼굴 인증 프로젝트 제품화 로드맵

이 문서는 현재 Unity C# 스크립트 형태의 얼굴 인증 프로토타입을 실제 제품 수준으로 발전시키기 위해 필요한 작업을 순서대로 정리한 문서입니다.

## 1. 현재 상태 확인

현재 프로젝트는 Unity에서 사용할 수 있는 얼굴 인증 로직의 초기 구조입니다.

- `FaceAuthenticationController`가 카메라 입력, 등록, 인증 흐름을 제어합니다.
- `MockFaceLandmarkProvider`는 실제 얼굴 인식이 아닌 테스트용 가짜 랜드마크를 생성합니다.
- `MediaPipeFaceLandmarkProvider`는 실제 MediaPipe 연동을 위한 자리만 마련되어 있으며, 아직 구현되지 않았습니다.
- 등록 정보는 `Application.persistentDataPath` 아래 `face_enrollment.json`으로 저장됩니다.
- 퍼지 로직 기반으로 등록된 사용자와 현재 얼굴 특징 벡터를 비교합니다.

제품화를 위해서는 실제 얼굴 랜드마크 추출, 보안 저장, 사용자 경험, 테스트, 배포 체계가 추가로 필요합니다.

## 2. Unity 프로젝트 정식 구성

1. Unity Hub에서 정식 프로젝트를 생성합니다.
2. 현재 `Assets/Scripts` 폴더를 Unity 프로젝트의 `Assets` 아래에 배치합니다.
3. `Scenes` 폴더를 만들고 기본 실행 씬을 생성합니다.
4. Canvas, Camera Preview, Status Text, 버튼 UI를 구성합니다.
5. `FaceAuthenticationController`를 빈 GameObject에 연결합니다.
6. Inspector에서 `cameraPreview`, `statusText` 값을 연결합니다.
7. Play Mode에서 Mock Provider 기준으로 등록 및 인증 흐름이 동작하는지 확인합니다.

## 3. 실제 얼굴 랜드마크 엔진 연동

현재 가장 중요한 미완성 작업은 `MediaPipeFaceLandmarkProvider` 구현입니다.

1. Unity용 MediaPipe Face Mesh 또는 동등한 얼굴 랜드마크 SDK를 선정합니다.
2. SDK를 Unity 프로젝트에 설치합니다.
3. 카메라 프레임을 SDK 입력 형식으로 변환합니다.
4. 얼굴 랜드마크 좌표와 confidence 값을 `FaceObservation`으로 변환합니다.
5. 얼굴이 감지되지 않는 경우, 여러 얼굴이 감지되는 경우, 일부 landmark가 가려지는 경우를 처리합니다.
6. `useMockProvider`를 false로 설정했을 때 실제 카메라 기반 인증이 동작하도록 만듭니다.

완료 기준:

- 실제 웹캠 영상에서 얼굴 landmark가 안정적으로 추출됩니다.
- `MediaPipeFaceLandmarkProvider.TryGetObservation`이 정상적으로 `true`를 반환합니다.
- Mock Provider 없이 등록과 인증이 가능합니다.

## 4. 사용자 등록 UX 구현

프로토타입은 키보드 입력으로 등록을 시작하지만, 제품에서는 명확한 UI가 필요합니다.

1. 사용자 등록 시작 버튼을 만듭니다.
2. 사용자 ID 또는 계정 정보를 입력받는 UI를 만듭니다.
3. 등록 진행률을 화면에 표시합니다.
4. 얼굴 위치 안내, 조명 안내, 정면 응시 안내를 추가합니다.
5. 등록 실패 사유를 사용자에게 명확히 보여줍니다.
6. 등록된 사용자 목록, 재등록, 삭제 기능을 추가합니다.

완료 기준:

- 사용자가 키보드 디버그 입력 없이 화면 UI만으로 등록할 수 있습니다.
- 등록 실패 시 원인을 알 수 있습니다.
- 등록된 사용자를 관리할 수 있습니다.

## 5. 인증 UX 구현

인증 흐름도 제품 수준으로 다듬어야 합니다.

1. 인증 시작, 인증 중, 성공, 실패 상태를 분리합니다.
2. 카메라 권한이 없거나 카메라를 찾을 수 없는 경우의 안내를 추가합니다.
3. 얼굴이 너무 멀거나 가까운 경우 안내를 추가합니다.
4. 가림, 고개 회전, 조명 부족 상태를 사용자에게 알려줍니다.
5. 인증 실패 횟수 제한과 재시도 정책을 정의합니다.
6. 인증 성공 후 연결될 실제 기능 또는 화면을 구현합니다.

완료 기준:

- 사용자가 현재 인증 상태를 즉시 이해할 수 있습니다.
- 실패했을 때 다음 행동을 알 수 있습니다.
- 인증 성공 이후 실제 서비스 흐름으로 이동합니다.

## 6. 보안 저장소 개선

현재 등록 데이터는 JSON 파일로 저장됩니다. 제품에서는 보안 저장 방식이 필요합니다.

1. 얼굴 원본 이미지 저장 여부를 명확히 결정합니다. 가능하면 원본 이미지는 저장하지 않습니다.
2. 저장되는 특징 벡터를 암호화합니다.
3. 플랫폼별 안전 저장소를 검토합니다.
   - Windows: DPAPI 또는 Credential Manager
   - macOS: Keychain
   - iOS: Keychain / Secure Enclave
   - Android: Keystore
4. 등록 데이터 무결성 검증을 추가합니다.
5. 데이터 삭제 기능을 구현합니다.
6. 개인정보 처리방침과 사용자 동의 절차를 준비합니다.

완료 기준:

- 등록 데이터가 평문 JSON으로 노출되지 않습니다.
- 사용자가 자신의 등록 데이터를 삭제할 수 있습니다.
- 개인정보 보호 요구사항을 문서화합니다.

## 7. 인증 정확도 개선

실제 제품에서는 오탐과 미탐을 줄이는 과정이 필요합니다.

1. 다양한 사용자, 조명, 각도, 거리 조건에서 테스트 데이터를 수집합니다.
2. 현재 특징 벡터 항목이 충분한지 검토합니다.
3. 사용자별 baseline과 sigma 계산 방식을 검증합니다.
4. 인증 threshold를 실험적으로 조정합니다.
5. 가림 상태에서 어떤 feature를 제외할지 정책을 정합니다.
6. False Accept Rate, False Reject Rate를 측정합니다.
7. 동일 인물 재인증과 타인 거부 테스트를 반복합니다.

완료 기준:

- 인증 성공률과 실패율을 수치로 확인할 수 있습니다.
- 제품에서 사용할 threshold 값이 근거를 갖습니다.
- 마스크, 안경, 조명 변화 등 주요 케이스를 테스트합니다.

## 8. 생체 인증 보안 강화

얼굴 인증 제품에는 위조 방지 대책이 필요합니다.

1. 사진, 영상, 화면 재생 공격에 대한 대응 방식을 정합니다.
2. 눈 깜박임, 고개 움직임, 랜덤 안내 동작 등 liveness check를 추가합니다.
3. depth camera 또는 infrared sensor 지원 여부를 검토합니다.
4. 공격 시나리오별 테스트 케이스를 만듭니다.
5. 인증 로그에 민감 정보를 남기지 않도록 정리합니다.

완료 기준:

- 단순 사진으로 인증이 통과하지 않습니다.
- 최소한의 liveness check가 구현됩니다.
- 공격 대응 한계가 문서화됩니다.

## 9. 코드 구조 정리

제품화 전에 유지보수 가능한 구조로 정리합니다.

1. `FaceAuthenticationController`의 책임을 UI, 카메라, 인증 흐름으로 분리합니다.
2. 설정값을 ScriptableObject 또는 별도 config로 분리합니다.
3. Provider 교체가 쉬운 구조를 유지합니다.
4. 에러 타입과 상태 타입을 enum으로 정리합니다.
5. Debug 전용 입력 코드를 개발 빌드 전용으로 제한합니다.
6. 네임스페이스 일관성을 정리합니다.

완료 기준:

- 핵심 인증 로직을 UI 없이 테스트할 수 있습니다.
- Provider를 교체해도 상위 흐름이 크게 바뀌지 않습니다.
- 설정값을 코드 수정 없이 조정할 수 있습니다.

## 10. 테스트 작성

제품 신뢰도를 위해 테스트를 추가합니다.

1. `FeatureVectorExtractor` 단위 테스트를 작성합니다.
2. `OcclusionAnalyzer` 단위 테스트를 작성합니다.
3. `EnrollmentManager` 저장, 로드, 등록, 삭제 테스트를 작성합니다.
4. `FuzzyFaceAuthenticator` 인증 성공과 실패 테스트를 작성합니다.
5. Mock Provider 기반 Play Mode 테스트를 작성합니다.
6. 실제 카메라 기반 수동 QA 체크리스트를 작성합니다.

완료 기준:

- 핵심 로직의 단위 테스트가 자동 실행됩니다.
- Unity Test Runner에서 테스트 결과를 확인할 수 있습니다.
- 수동 QA 체크리스트가 문서화됩니다.

## 11. 빌드 및 배포 준비

1. Unity Build Settings에서 대상 플랫폼을 정합니다.
2. Windows, macOS, Android, iOS 중 1차 배포 플랫폼을 선택합니다.
3. 카메라 권한 설정을 플랫폼별로 추가합니다.
4. 앱 이름, 아이콘, 버전 정보를 설정합니다.
5. Development Build와 Release Build 설정을 분리합니다.
6. 빌드 결과물을 실제 기기에서 실행합니다.
7. 설치, 실행, 권한 허용, 등록, 인증, 종료 흐름을 확인합니다.

완료 기준:

- 선택한 1차 플랫폼에서 설치 가능한 빌드가 생성됩니다.
- 카메라 권한 요청과 인증 흐름이 실제 환경에서 동작합니다.
- 릴리스 빌드에서 디버그 UI와 로그가 정리됩니다.

## 12. GitHub 관리

GitHub 저장소에는 코드뿐 아니라 실행 방법과 개발 상태를 함께 관리해야 합니다.

1. `README.md`를 작성합니다.
2. 프로젝트 소개, 주요 기능, 설치 방법, 실행 방법을 정리합니다.
3. Unity 버전과 필요한 패키지를 명시합니다.
4. 현재 Mock Provider와 실제 Provider의 차이를 설명합니다.
5. 이 로드맵 문서를 링크합니다.
6. `.gitignore`에 Unity 불필요 파일을 추가합니다.
7. 기능 단위로 branch를 나누어 개발합니다.
8. 주요 변경사항은 commit message에 명확히 남깁니다.

완료 기준:

- 처음 보는 개발자가 README만 보고 프로젝트를 실행할 수 있습니다.
- GitHub에 제품화 진행 상황이 문서화됩니다.
- 불필요한 Unity 캐시 파일이 저장소에 올라가지 않습니다.
